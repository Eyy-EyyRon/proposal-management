Command & Control
User Manual

A tiered delegation system that lets the CEO focus on high-value conversations while the team handles proposal creation, verification, and client communications with full traceability — including inline urgency control, CEO push-back loops, department-scoped boards, signed PDF certificates, and cryptographic audit seals.

1. The Command & Control Workflow

A four-layer delegation chain. Every CEO-tasked proposal is drafted, verified, and polished before the CEO ever sees it.

STEP 1
👑

CEO Creates Task

Sets client, urgency (P1–P3), template & assigns to General Admin.

STEP 2
🛡

Admin Routes

Assigns to Dept Admin who picks the Staff member to build the draft.

STEP 3
⚙️

Staff Builds & Dept Admin Verifies

Staff drafts, submits. Dept Admin approves or requests revision.

STEP 4
✅

CEO Talks & Sends

Verified proposal appears in Talking Inbox. CEO reviews & sends to client.

3. Urgency Tiers & SLA Timers

Every task gets an urgency level when the CEO creates it. The SLA timer counts down in real-time across all dashboards.

P1

Critical Priority

⏱ 2-hour SLA

Reserved for the most time-sensitive opportunities. Entire UI enters focus mode.

UI: Red pulse ring + animated "CRITICAL TASK" banner with live countdown at top of proposal.
Task Board: Pinned to top of all boards in rose section with animated ring around each card.
Alert: Toast fires when <30 min remaining. Cannot be missed.
P2

High Priority

⏱ 24-hour SLA

Important deals requiring same-day attention. Highlighted in orange throughout the interface.

UI: Orange border on task cards. Countdown turns amber when <6 hours remain.
Task Board: Standard order, orange-bordered card.
Alert: Live countdown visible on all related views.
P3

Standard Priority

⏱ 72-hour SLA

Regular pipeline proposals. Grey countdown badge, no special alerts.

⚠ SLA Breach

When the timer hits zero, the badge turns red and shows "OVERDUE by Xm". The task remains fully actionable — SLA breach is a tracking indicator, not a lock.

🔐 Super Admin Guide — Elevated Operator

Super Admins have access to system-wide operations — but every destructive action requires a temporary, time-limited JIT Elevation. No standing privileges. Every action is logged.

Your Dashboard

Navigate to /super-admin. The dashboard shows system-wide overview, all users, all proposals, and a team management panel. Sensitive actions (deactivate user, change role) are locked until you request elevation.

Requesting JIT Elevation

1

Click any locked (greyed) action button

A lock badge appears on guarded buttons. Clicking it opens the Elevation Request modal automatically.

2

Select Tier & Duration

Operational (auto-approved, 30 min–4 h): for deactivation, department assignment. Critical (requires CEO approval): for role changes promoting to Super Admin.

3

Write a Justification (min 10 characters)

Required. Attached to every action audit log entry for the duration of the session.

4

Active Banner Appears

An orange (Operational) or red (Critical) pulsing banner appears at the top of every page for the session duration. The live countdown is always visible.

5

Confirm Each Destructive Action

Every guarded button opens a JIT Guard confirmation dialog: state a per-action reason (min 10 chars), then click Confirm & Execute.

ActionElevation RequiredCEO Notified?
Deactivate a userOperationalNo
Assign departmentOperationalNo
Change user roleCritical (CEO approval)Yes — approval request sent
Access audit logNone requiredNo

Ghost Session Protection

If the CEO triggers the Emergency Brake, your session is force-terminated immediately — even mid-action. All Firestore write attempts from a ghost session are rejected at the security rules level via the notForcedOut() guard. You will be redirected to the login page with a security reset notice.

Elevation Auto-Expiry

When the countdown reaches zero, elevation is revoked automatically. Locked buttons return to their greyed state. Any in-flight destructive action will be rejected by Firestore rules once the elevation document expires.

🔒 Audit Trail & Signed Certificates

Every accepted proposal generates a tamper-proof audit record with a cryptographic hash, signer IP, and downloadable certificate.

🔑 Content Hash

SHA-256 hash of the proposal HTML computed at sign-time. Shown on the CEO proposal detail page with a green Hash Verified badge — or a red Tampered alert if the document was altered post-signing.

🌐 Signer IP

Client's IP address captured via /api/get-signer-ip at the moment of signature. Stored in the audit seal and visible to CEO & Admin on the proposal detail page.

📜 Signed Certificate

Click Download Certificate on the CEO proposal detail page to generate a signed HTML snapshot via /api/generate-signed-pdf. Includes proposal content, signer name, timestamp, IP, and hash.

Hash Tamper Alert

If the proposal content is edited after the client has signed, the CEO detail page shows a red ⚠ Document Tampered banner. The audit seal is immutable — it records what the client actually signed, not the current state of the document.

Client Thank-You Page

After signing, clients see a confirmation screen with their audit seal reference. A Download Signed Copy button generates the same certificate for the client's own records.

⚡ Inline Urgency Editor

Any member of the delegation chain — CEO, Admin, Dept Admin, or Staff — can change a task's urgency at any time directly from the task card. No modal required.

How It Works

  1. Find the task card on your board (Task Center, Task Board, or My Tasks).
  2. Click the coloured urgency badge (e.g. P3 Normal) — a dropdown appears.
  3. Select the new urgency tier. The SLA timer resets from now using the new window.
  4. The change is saved to Firestore instantly and logged in task history with actor name and timestamp.
From → ToEffectWho Is Notified
Any → P1SLA resets to 2h · red pulse ring · CEO notifiedCEO (if not the actor)
Any → P2SLA resets to 24h · orange borderNo additional notification
Any → P3SLA resets to 72h · standard greyNo additional notification

Urgency changes are not reversible via undo — but every change is recorded in the task history audit log with the actor's name. Escalating frequently can cause SLA timer resets and should only be done when genuinely warranted.

🔐 Just-In-Time (JIT) Elevation System

The JIT system enforces the principle of least privilege. Super Admins have no standing destructive access — they must formally request it, justify it, and accept a time limit. Every elevated session is fully audited.

⚡ Operational Tier

  • Auto-approved immediately
  • Duration: 30 min, 1 hour, or 4 hours
  • Unlocks: user deactivation, department assignment
  • Orange pulsing banner shown during session
  • CEO notified in Security Feed

🔴 Critical Tier

  • Requires CEO approval before activation
  • Amber "Pending" banner shown while waiting
  • CEO receives in-app notification to approve/deny
  • Unlocks: role changes (promotes to Super Admin)
  • Red pulsing banner once approved

The Elevation Modal

  1. Click a locked action button — the Elevation Request modal opens.
  2. Select Operational or Critical tier.
  3. Choose duration (30 min / 1 h / 4 h).
  4. Enter a justification (minimum 10 characters — enforced).
  5. Click Request Access. Operational activates instantly; Critical enters pending state.
⚠ Safety notice displayed in modal: "All actions performed during this session will be logged and tied to your IP address."

The Active Power Banner

Once elevation is active, a persistent banner appears at the top of every page for the session duration:

Active Admin Mode  ·  Operational  ·  47m 12s remaining  
Critical Admin Mode  ·  CEO-Approved  ·  23m 05s remaining  

The JIT Guard — Destructive Action Wrapper

Every destructive button (Deactivate, Change Role) is wrapped in a JIT Guard. When elevation is not active:

🔒 Locked State

Button is greyed out with a 🔒 Elevate or 🔒 Critical badge overlay. Clicking opens the elevation modal — not a tooltip, a real prompt.

✅ Elevated State

Button is fully clickable. On click, a per-action confirmation dialog appears requiring a separate reason (min 10 chars) and Confirm & Execute.

Audit Log Integration

Every jit_elevation_requested, jit_elevation_approved, jit_elevation_denied, and jit_elevation_revoked event is recorded in the logs collection and visible on the Audit Log page at /dashboard/audit-log.

🚨 Emergency Brake — Nuclear Option

The CEO's last line of defence. One action simultaneously revokes all Super Admin elevation sessions, invalidates their Firebase auth tokens, force-logs them out of every device, and writes an immutable audit record.

What It Does (in one atomic operation)

1.Wipes all documents in the elevations collection
2.Sets forceLogoutTimestamp on every Super Admin user document
3.Revokes Firebase auth tokens — ends all active sessions on all devices
4.Writes an immutable Black Box record to system_purge_logs with IP + User Agent captured
5.Writes an emergency_brake_activated entry to the standard logs audit collection
6.Returns a logId shown immediately in the CEO's success toast

How to Trigger

1

Navigate to /ceo-dashboard/security

Click Emergency Brake (red button, top-right).

2

Read the consequences dialog

Lists how many active sessions will be terminated.

3

Type REVOKE and click Execute Emergency Brake

The confirmation word prevents accidental triggers. This cannot be undone.

Security Sentry — Client-Side Detection

Every Super Admin's browser continuously watches their own user document. When forceLogoutTimestamp is updated to a time after their session started, the Security Sentry triggers an automatic signout and redirects to the login page with a ?reason=security_reset banner.

Ghost Session Guard (Firestore Rules)

Even if a Super Admin has an active session that isn't caught client-side, all Firestore write operations check notForcedOut() in security rules. Any write from an auth token issued before the forceLogoutTimestamp is silently rejected at the database level.

📦 Black Box — System Purge Logs

Every Emergency Brake event writes an immutable, append-only record to a dedicated system_purge_logs Firestore collection. No one — not even the CEO — can edit or delete these records.

🌐 Network Metadata

CEO's IP address (captured from request headers) and full User Agent string recorded at the exact moment of trigger.

🔢 Scope Data

Number of admins revoked, number of elevation sessions wiped, list of affected UIDs — all stored in the log document.

🔗 Cross-Reference

The standard logs audit entry references the purgeLogId, linking the general audit trail to the black box record.

Viewing Purge Logs (CEO Only)

Navigate to /ceo-dashboard/security/logs or click Purge Logs in the Security Monitor header or CEO sidebar. Each log entry shows:

FieldDescription
Log IDUnique Firestore document ID — shown in success toast immediately after brake fires
TimestampExact server time of the purge event
ActorCEO's name and UID who triggered the brake
IP AddressCEO's IP at time of trigger (expandable detail panel)
User AgentBrowser/OS string (expandable detail panel)
Revoked CountNumber of Super Admins whose tokens were revoked
Elevations WipedNumber of active elevation documents deleted
Affected UIDsList of Super Admin user IDs who were force-logged out

Immutability Enforced at Database Level

Firestore security rules on system_purge_logs allow create only (CEO, via Cloud Function) and read (CEO only). update and delete return false for every user including the CEO. These records cannot be altered by anyone after creation.

Notifications for JIT Elevation (CEO)
When a Super Admin requests Critical elevation, an in-app notification appears on the CEO's notification bell. Click Approve or Deny directly from the notification card — no need to navigate to the Security Monitor.
